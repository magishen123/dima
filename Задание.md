# Модуль 1
Произведите базовую настройку устройств:

1. Настройте имена устройств согласно топологии. Используйте полное доменное имя
2. На всех устройствах необходимо сконфигурировать IPv4:
    2.1. IP-адрес должен быть из приватного диапазона в случае, если сеть локальная, согласно RFC1918
    2.2. Локальная сеть в сторону HQ-SRV должна вмещать не более 32 адресов
    2.3. Локальная сеть в сторону HQ-CLI должна вмещать не менее 16 адресов
    2.4. Локальная сеть для управления должна вмещать не более 8 адресов
    2.5. Локальная сеть в сторону BR-SRV должна вмещать не более 16 адресов
3. Настройте доступ к сети Интернет, на маршрутизаторе ISP:
    3.1. Настройте адресацию на интерфейсах:
        3.1.1. Интерфейс, подключенный к магистральному провайдеру, получает адрес по DHCP
        3.1.2. Настройте маршрут по умолчанию, если это необходимо
        3.1.3. Настройте интерфейс, в сторону HQ-RTR, интерфейс подключен к сети 172.16.1.0/28
        3.1.4. Настройте интерфейс, в сторону BR-RTR, интерфейс подключен к сети 172.16.2.0/28
    3.2. На ISP настройте динамическую сетевую трансляцию портов для доступа к сети Интернет HQ-RTR и BR-RTR.
    3.3. Создайте локальные учетные записи на серверах HQ-SRV и BR-SRV:
        3.3.1. Создайте пользователя sshuser
        3.3.2. Пароль пользователя sshuser с паролем P@ssw0rd
        3.3.3. Идентификатор пользователя 2026
        3.3.4. sshuser должен иметь возможность запускать sudo без ввода пароля
    3.4. Создайте пользователя net_admin на маршрутизаторах HQ-RTR и BR-RTR
    3.5. Пароль пользователя net_admin с паролем P@ssw0rd
    3.6. При настройке ОС на базе Linux, запускать sudo без ввода пароля
    3.7. При настройке ОС отличных от Linux пользователь должен обладать максимальными привилегиями.
4. Реализовать на HQ-RTR маршрутизацию трафика
5. безопасный удаленный доступ на серверах HQ-SRV и BR- SRV
    5.1. Для подключения используйте порт 2026
    5.2. Разрешите подключения исключительно пользователю sshuser
    5.3. Ограничьте количество попыток входа до двух
    5.4. Настройте баннер «Authorized access only».
6. Между офисами HQ и BR, на маршрутизаторах HQ-RTR и BR-RTR необходимо сконфигурировать ip туннель:
    6.1.1. На выбор технологии GRE или IP in IP
    6.1.2. Сведения о туннеле занесите в отчёт.
7. Обеспечьте динамическую маршрутизацию на маршрутизаторах HQ-RTR и BR-RTR: сети одного офиса должны быть доступны из другого офиса и наоборот. Для обеспечения динамической маршрутизации используйте link state протокол на усмотрение участника:
    7.1. Разрешите выбранный протокол только на интерфейсах ip туннеля
    7.2. Маршрутизаторы должны делиться маршрутами только друг с другом
    7.3. Обеспечьте защиту выбранного протокола посредством парольной защиты
    7.4. Сведения о настройке и защите протокола занесите в отчёт.
8. Настройка динамической трансляции адресов маршрутизаторах HQ- RTR и BR-RTR:
    8.1. Настройте динамическую трансляцию адресов для обоих офисов в сторону ISP, все устройства в офисах должны иметь доступ к сети Интернет
    8.2. протокол динамической конфигурации хостов для сети в сторону HQ-CLI:
        8.2.1. Настройте нужную подсеть
        8.2.2. В качестве сервера DHCP выступает маршрутизатор HQ-RTR
        8.2.3. Клиентом является машина HQ-CLI
        8.2.4. Исключите из выдачи адрес маршрутизатора
        8.2.5. Адрес шлюза по умолчанию – адрес маршрутизатора HQ-RTR
        8.2.6. Адрес DNS-сервера для машины HQ-CLI – адрес сервера HQ-SRV
        8.2.7. DNS-суффикс – sirius-exam.org
9. Настройте инфраструктуру разрешения доменных имён для офисов HQ и BR:
    9.1. Основной DNS-сервер реализован на HQ-SRV
    9.2. Сервер должен обеспечивать разрешение имён в сетевые адреса устройств и обратно в соответствии с таблицей 3
    9.3. В качестве DNS сервера пересылки используйте любой общедоступный DNS сервер (77.88.8.7, 77.88.8.3 или другие)
    9.4. Настройте часовой пояс на всех устройствах (за исключением виртуального коммутатора, в случае его использования) согласно месту проведения экзамена

# Модуль 2
1. Установите и настройте службу Samba на сервере BR-SRV:
    1.1. Дайте машине имя в формате FQDN. Имя домена sirius-exam.org
    1.2. Предоставьте машине HQ-CLI доступ к BR-SRV
    1.3. Создайте 5 пользователей для офиса HQ: имена пользователей формата hquser№ (например, hquser1, hquser2 и т.д.)
    1.4. Создайте группу hq, введите в группу созданных пользователей
    1.5. Убедитесь, что пользователи группы hq имеют право аутентифицироваться на HQ-CLI
2. Сконфигурируйте файловое хранилище на сервере HQ-SRV:
    2.1. При помощи двух подключенных к серверу дополнительных дисков размером 1 Гб сконфигурируйте дисковый массив уровня 0
    2.2. Имя устройства – md0, при необходимости конфигурация массива размещается в файле /etc/mdadm.conf
    2.3. Создайте раздел, отформатируйте раздел, в качестве файловой системы используйте ext4
    2.4. Обеспечьте автоматическое монтирование в папку /raid
3. Настройте сервер сетевой файловой системы (nfs) на HQ-SRV:
    3.1. В качестве папки общего доступа выберите /raid/nfs, доступ для чтения и записи исключительно для сети в сторону HQ-CLI
    3.2. На HQ-CLI настройте автомонтирование в папку /mnt/nfs
    3.3. Основные параметры сервера отметьте в отчёте
4. Настройте службу сетевого времени на базе сервиса chrony на маршрутизаторе ISP:
    4.1. Вышестоящий сервер ntp на маршрутизаторе ISP - на выбор участника
    4.2. Стратум сервера – 5
    4.3. В качестве клиентов ntp настройте: HQ-SRV, HQ-CLI, BR-RTR, BR-SRV.
5. Разверните веб приложение в docker на сервере BR-SRV:
    5.1. Средствами docker должен создаваться стек контейнеров с веб приложением и базой данных
    5.2. Используйте образы site_latest и mariadb_latest располагающиеся в директории docker в образе Additional.iso
    5.3. Основной контейнер testapp должен называться tespapp
    5.4. Контейнер с базой данных должен называться db
    5.5. Импортируйте образы в docker, укажите в yaml файле параметры подключения к СУБД, имя БД - testdb, пользователь testс паролем P@ssw0rd, порт приложения 8080, при необходимости другие параметры
    5.6. Приложение должно быть доступно для внешних подключений через порт 8080
6. Разверните веб приложение на сервере HQ-SRV:
    6.1. Используйте веб-сервер apache
    6.2. В качестве системы управления базами данных используйте mariadb
    6.3. Файлы веб приложения и дамп базы данных находятся в директории web образа Additional.iso
    6.4. Выполните импорт схемы и данных из файла dump.sql в базу данных webdb
    6.5. Создайте пользователя webс паролем P@ssw0rd и предоставьте ему права доступа к этой базе данных
    6.6. Файлы index.php и директорию images скопируйте в каталог веб сервера apache
    6.7. В файле index.php укажите правильные учётные данные для подключения к БД
    6.8. Запустите веб сервер и убедитесь в работоспособности приложения
    6.9. Основные параметры отметьте в отчёте
7. На маршрутизаторах сконфигурируйте статическую трансляцию портов:
    7.1. Пробросьте порт 8080 в порт приложения testapp BR-SRV на маршрутизаторе BR-RTR, для обеспечения работы приложения testapp извне
    7.2. Пробросьте порт 8080 в порт веб приложения на HQ-SRV на маршрутизаторе HQ-RTR, для обеспечения работы веб приложения извне
    7.3. Пробросьте порт 2026 на маршрутизаторе HQ-RTR в порт 2026 сервера HQ-SRV, для подключения к серверу по протоколу ssh из внешних сетей
    7.4. Пробросьте порт 2026 на маршрутизаторе BR-RTR в порт 2026 сервера BR-SRV, для подключения к серверу по протоколу ssh из внешних сетей.
8. Настройте веб-сервер nginx как обратный прокси-сервер на ISP
    8.1. При обращении по доменному имени web.sirius-exam.org у клиента должно открываться веб приложение на HQ-SRV
    8.2. При обращении по доменному имени docker.sirius-exam.org клиента должно открываться веб приложение testapp
9. На маршрутизаторе ISP настройте web-based аутентификацию:
    9.1. При обращении к сайту web.sirius-exam.org клиенту должно быть предложено ввести аутентификационные данные
    9.2. В качестве логина для аутентификации выберите WEBс паролем P@ssw0rd
    9.3. Выберите файл /etc/nginx/.htpasswd в качестве хранилища учётных записей
    9.4. При успешной аутентификации клиент должен перейти на веб сайт.
10. Удобным способом установите приложение Яндекс Браузер на HQ-CLI

# Модуль 3. Выполнение работ по проектированию контейнерной инфраструктуры

Требуется развернуть контейнерную инфраструктуру для облачного хранилища Nextcloud на BR-SRV (ОС РедОС).

1. Необходимо установить Docker и Docker Compose, создать рабочий каталог /opt/nextcloud, разработать файл docker-compose.yml с двумя сервисами (MySQL и Nextcloud).
2. Настроить volumes для обоих контейнеров из созданного каталога. Запустить контейнеры. Установить Nginx, создать самоподписанный SSL-сертификат.
3. Настроить конфигурацию Nginx для проксирования запросов на порт 8080 контейнера Nextcloud с передачей заголовков X-Real-IP и X-Forwarded-Proto, а также заголовка HSTS.
4. Создать символическую ссылку в директории /etc/nginx/sites-enabled/.
5. Выполнить проверку работоспособности через браузер по адресу https://127.0.0.1

На HQ-CLI:

1. Требуется создать группы Work, Job и labor.
2. Создать пользователей User1, User2 в группе Work, User3, User4 в группе Job, User5 в группе labor.
3. Создать каталог /home/Folder. Внутри него создать подкаталог /home/Folder/work_shared с правами 770 (владелец root, группа Work).
4. Создать подкаталог /home/Folder/job_readonly с правами 750 (владелец root, группа Job).
5. Зафиксировать IP-адрес Сервера 1.


На HQ-CLI:

1. Открыть браузер, перейти к Nextcloud по HTTPS, подтвердить переход по самоподписанному сертификату, убедиться в открытии страницы установки Nextcloud.
2. Выполнить проверку прав: от User1 — создание и удаление файла в work_shared; от User3 — чтение и невозможность записи в job_readonly; от User5 — отсутствие доступа к обоим подкаталогам.
3. На BR-SRV запустить просмотр логов Nginx и убедиться в появлении IP-адреса Сервера 2.
4. На HQ-CLI выполнить проверку заголовка HSTS через curl и убедиться в наличии строки Strict-Transport-Security: max-age=31536000; includeSubDomains; preload.
